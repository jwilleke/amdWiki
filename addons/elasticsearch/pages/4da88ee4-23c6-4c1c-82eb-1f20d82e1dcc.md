# External Asset Search — Administrator Guide

This page covers configuration, role-based access control, and operational tasks for the Elasticsearch external asset provider.

## Overview

The elasticsearch addon bridges a [sist2](https://github.com/simon987/sist2) external asset index (NAS, S3, or any compatible storage) into ngdpbase's AssetManager. Editors and contributors can search and insert indexed files from within the wiki editor without any file copying.

**Architecture:**

```
NAS filesystem → sist2 scanner → Elasticsearch index → ngdpbase AssetProvider → asset browser
```

Thumbnails are proxied through ngdpbase — browsers never contact the sist2 host directly.

## Configuration

Add the following keys to your instance `app-custom-config.json`:

```json
"ngdpbase.addons.elasticsearch.enabled": true,
"ngdpbase.addons.elasticsearch.es-url":    "http://192.168.68.71:9200",
"ngdpbase.addons.elasticsearch.es-index":  "sist2",
"ngdpbase.addons.elasticsearch.sist2-url": "http://192.168.68.71:4090",
"ngdpbase.addons.elasticsearch.index-ids": [],
"ngdpbase.addons.elasticsearch.path-access": {
  "admin":       [],
  "editor":      ["jims/", "family/", "shared/"],
  "contributor": ["shared/"]
}
```

### Config Keys

| Key | Default | Description |
|-----|---------|-------------|
| `enabled` | `false` | Enable the addon |
| `es-url` | `http://localhost:9200` | Elasticsearch base URL |
| `es-index` | `sist2` | Elasticsearch index name |
| `sist2-url` | `http://localhost:4090` | sist2 search UI URL (used for file and thumbnail serving) |
| `index-ids` | `[]` | Array of sist2 scan index IDs to filter on. Empty = all indices |
| `path-access` | absent | Role → allowed path prefixes map (see below) |

## Principal-Based Path Access

The `path-access` key controls which folder/path prefixes each principal can see in search results. Keys are **principals** — either role names or usernames — consistent with the `audience` / `access` front matter model used for page access control.

```json
"ngdpbase.addons.elasticsearch.path-access": {
  "admin":       [],
  "jim":         ["jims/", "family/", "mjs/", "molly/", "shared/"],
  "editor":      ["family/", "shared/"],
  "contributor": ["shared/"],
  "viewer":      ["public/"]
}
```

**Rules:**

- An **empty array `[]`** means unrestricted — that principal sees all paths
- **Keys are principals**: use a role name (`admin`, `editor`) or a specific username (`jim`) — same semantics as `audience` in page front matter
- **Multiple matches are unioned**: a user `jim` with role `editor` sees `jims/ ∪ family/ ∪ shared/`
- A principal **not listed** in the map falls through to unrestricted
- **Absent `path-access` key** entirely = no filtering (all authorised users see everything)

The route already requires `editor`, `contributor`, or `admin` to access the asset browser — unauthenticated users cannot reach the asset search at all.

### NAS Top-Level Paths (jimstest)

| Path | Contents |
|------|----------|
| `jims/` | Personal files |
| `family/` | Shared family files |
| `mjs/` | MJS files |
| `molly/` | Molly's files |
| `shared/` | General shared storage |

## Triggering a Manual Scan

The sist2 scheduler runs at `0 0 * * *` (midnight daily). To trigger an immediate scan:

**Incremental scan** (new/changed files only — fast):

```
GET http://192.168.68.71:9092/api/job/NAS%20Scan/run
```

**Full scan** (rescans everything — required if mounts were unavailable during a previous scan):

```
GET http://192.168.68.71:9092/api/job/NAS%20Scan/run?full=true
```

Monitor progress at the sist2 admin UI: `http://192.168.68.71:9092/ui/#/job/NAS%20Scan`

## NAS Mount Health

The NAS folders are automounted on deby. If a mount is idle for too long the OS unmounts it, causing sist2 to miss those folders during a scan.

**Fix:** Set `TimeoutIdleSec=0` in each automount unit file on deby:

```ini
[Automount]
Where=/host/family
TimeoutIdleSec=0
```

After editing the unit files:

```bash
sudo systemctl daemon-reload
sudo systemctl restart host-family.automount
```

This was applied to all five mount units (family, mjs, molly, shared, jims) in April 2026.

## Health Check

The addon reports its health on the `/admin/addons` panel. It calls `GET /i` on the sist2 UI URL. Status shows:

- **Healthy** — sist2 is reachable and responding
- **Unhealthy** — check `es-url` and `sist2-url` config; verify the sist2 container is running

## sist2 Index IDs

Each sist2 scan creates an index with a unique integer ID. Use `index-ids` to restrict results to specific scans:

```json
"ngdpbase.addons.elasticsearch.index-ids": [1776030791]
```

To find available index IDs:

```
GET http://192.168.68.71:4090/i
```

The `indices` array lists all available scans with their `id`, `name`, and `root` path.

## Limitations

- **Read-only**: files cannot be uploaded to the NAS through the wiki
- **Tag write-back**: sist2 tags are exposed as keywords but cannot be modified from the wiki (deferred to a future release)
- **Thumbnail availability**: thumbnails are generated by sist2 progressively; newly indexed files may lack thumbnails until the next scan cycle
