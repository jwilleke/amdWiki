## Overview

Private pages are wiki pages that are visible only to their **creator** and to **administrators**. All other users — including other authenticated users — are denied access and receive a 403 response when they attempt to view, edit, or navigate to the page.

Private pages are stored in a separate location on disk (`pages/private/{creator}/`) and are excluded from search results for users who do not have permission to see them.

### Key facts at a glance

| Property | Value |
|----------|-------|
| Access | Creator + Admins only |
| Search visibility | Creator + Admins see results; others do not |
| Attachments | Also stored privately |
| Required pages | **Cannot** be made private |
| Storage | `pages/private/{creator}/{uuid}.md` |

## Marking a Page as Private

A page becomes private when you add the **`private`** user keyword to it.

### Creating a new private page

1. Click **New Page** or navigate to a page name that does not exist yet.
2. In the **Keywords** field (on the edit form), type `private`.
3. Write your content and click **Save**.

The page is immediately stored in the private area and is inaccessible to other users.

### Making an existing page private

1. Open the page and click **Edit**.
2. In the **Keywords** field, add `private` to the list of keywords.
3. Click **Save**.

The page file is moved to the private storage area on save. Its URL, UUID, and page name remain unchanged.

### Removing the private keyword

1. Open the page and click **Edit**.
2. Remove `private` from the **Keywords** field.
3. Click **Save**.

The page is moved back to the standard pages area and becomes visible to all users who have access to the wiki.

## Who Can See a Private Page

| User type | Can view? | Can edit? | Can delete? |
|-----------|-----------|-----------|-------------|
| The page creator | Yes | Yes | Yes |
| Administrators | Yes | Yes | Yes |
| Other authenticated users | No (403) | No (403) | No (403) |
| Anonymous visitors | No (403) | No | No |

### Search results

Private pages appear in search results **only** for their creator and for administrators. No metadata, title, or excerpt is revealed to other users.

## Private Attachments

Files attached to a private page are stored privately and subject to the same access rules as the page itself.

- When you upload an attachment to a private page, the file is stored in `attachments/private/{creator}/` rather than the standard attachments folder.
- Attempting to access the attachment URL directly (e.g. `/attach/MyPrivatePage/document.pdf`) returns **403 Forbidden** for anyone who does not have access to the parent page.
- If the page is later made public (by removing the `private` keyword), existing attachments **remain** in the private storage area until they are re-uploaded. Download links will continue to be protected.

## Limitations

### Required pages cannot be private

Pages in the **required-pages** category (system documentation, wiki help pages, etc.) are committed to the project repository and are always publicly visible within the wiki. Attempting to add the `private` keyword to a required page returns an error:

> *Required pages cannot be marked as private.*

### Sharing with specific users

The current implementation is all-or-nothing: a private page is visible only to its creator and admins. There is no mechanism to share a private page with a specific set of non-admin users. If you need to share content with a subset of users, consider using a role-based access approach or the standard wiki (all authenticated users can view).

### Bulk operations

The **Page Index** admin tool shows private pages only for admins. Bulk export and import operations respect the private flag.

## Access Denied Behaviour

When a user who is not the creator or an admin tries to reach a private page, [{$applicationname}] responds with **HTTP 403 Forbidden**. This applies to:

- `/view/{PageName}` — view
- `/edit/{PageName}` — edit
- `/delete/{PageName}` — delete
- `/history/{PageName}` — page history
- `/attach/{PageName}/{filename}` — attachments

The 403 response is identical to the response for a page that does not exist, so that the existence of a private page is not revealed to unauthorised users.

## Tips and Best Practices

- **Use private pages for drafts.** Write content privately and remove the `private` keyword when you are ready to publish.
- **Admin review.** Administrators can always see and edit private pages. This ensures that no content can be permanently hidden from site management.
- **Naming.** Private page names follow the same rules as regular page names. Choose names that make sense to you — they are not visible to other users.
- **Attachments first.** If you plan to make a page public, re-upload any attachments after removing the `private` keyword, so they move to the public attachments area.
