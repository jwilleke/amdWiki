# User Keywords

This page defines the user-extensible vocabulary for flexible content tagging and organization. User keywords are **configuration-driven** and defined in `config/app-default-config.json` under `amdwiki.userKeywords`.

[{$pagename}] is part of [Page Metadata Documentation]

## Configuration Structure

User keywords are defined in `config/app-default-config.json`:

```json
{
  "amdwiki.userKeywords": {
    "default": {
      "label": "default",
      "description": "Default keyword with no special restrictions",
      "category": "general",
      "enabled": true,
      "restrictEditing": false
    },
    "private": {
      "label": "private",
      "description": "Private content editable only by author and admins",
      "category": "access",
      "enabled": true,
      "restrictEditing": true,
      "allowedRoles": ["admin", "author"]
    }
  }
}
```

### Configuration Properties

Each user keyword has the following properties:

| Property | Type | Description |
|----------|------|-------------|
| `label` | string | The keyword value used in metadata `user-keywords` array |
| `description` | string | Human-readable description of the keyword's purpose |
| `category` | string | Grouping category (e.g., "general", "access", "status", "subject") |
| `enabled` | boolean | Whether the keyword is active and available for use |
| `restrictEditing` | boolean | If true, restricts who can edit pages with this keyword |
| `allowedRoles` | array | Roles allowed to edit (only used if `restrictEditing` is true) |
| `schemaOrg` | string | Optional schema.org type mapping for semantic web |

## Current User Keywords

### General Keywords

* **default** - Default keyword with no special restrictions
  * Category: `general`
  * No restrictions
  * Use for: Standard content without special needs

### Access Control Keywords

* **private** - Private content editable only by author and admins
  * Category: `access`
  * Restricts editing to: author, admin
  * Use for: Personal notes, drafts, sensitive content

### Status Keywords

* **draft** - Work in progress content
  * Category: `status`
  * Use for: Incomplete pages, early-stage content

* **review** - Content under review
  * Category: `status`
  * Use for: Content awaiting approval or feedback

* **published** - Finalized and approved content
  * Category: `status`
  * Use for: Completed, reviewed, approved pages

### Subject Keywords

* **medicine** - Medical and healthcare-related content
  * Category: `subject`
  * Schema.org: `MedicalEntity`

* **geology** - Geological and earth science content
  * Category: `subject`

* **meteorology** - Weather and atmospheric science content
  * Category: `subject`

* **oceanography** - Ocean and marine science content
  * Category: `subject`

* **psychology** - Psychological and behavioral science content
  * Category: `subject`

* **sociology** - Social science and society-related content
  * Category: `subject`

* **anthropology** - Human culture and society content
  * Category: `subject`

* **economics** - Economic and financial content
  * Category: `subject`

* **political-science** - Political science and governance content
  * Category: `subject`


## Usage in Page Metadata

User keywords are used in the `user-keywords` frontmatter field as an array:

```yaml
---
title: Example Medical Research Page
system-category: documentation
user-keywords:
  - medicine
  - research
  - published
slug: medical-research-example
---
```

### Metadata Rules

* **Maximum Keywords**: Up to 5 keywords per page (configurable via `amdwiki.maximum.user-keywords`)
* **Case Sensitivity**: Keywords are case-insensitive
* **Array Format**: Must be YAML array format
* **Validation**: Warnings for keywords not in configuration (but still allowed)

## User Keywords vs System Categories

| Aspect | System Categories | User Keywords |
|--------|------------------|---------------|
| **Field** | `system-category` | `user-keywords` |
| **Format** | Single value (string) | Multiple values (array) |
| **Purpose** | Storage location control | Content tagging & classification |
| **Effect** | Determines `/pages/` or `/required-pages/` | No storage effect, used for search/organization |
| **Configuration** | `amdwiki.systemCategories` | `amdwiki.userKeywords` |
| **Maximum** | 1 required | Up to 5 optional |
| **Extensibility** | Admin-controlled | User-extensible (Editor+ can propose) |

### Example Usage

```yaml
---
title: Introduction to Oceanography
system-category: documentation  # ← Single category (WHERE to store)
user-keywords:                   # ← Multiple keywords (WHAT it's about)
  - oceanography
  - geology
  - published
slug: intro-oceanography
---
```

## Keyword Categories

User keywords are organized into categories for better management:

### 1. General (`general`)
* `default` - Standard content

### 2. Access Control (`access`)
* `private` - Author and admin only
* `confidential` - Admin only (custom)
* `restricted` - Limited access (custom)

### 3. Status (`status`)
* `draft` - Work in progress
* `review` - Under review
* `published` - Finalized content
* `archived` - Historical content (custom)

### 4. Subject (`subject`)
* Science: `medicine`, `geology`, `meteorology`, `oceanography`, `psychology`, `chemistry`, `physics`
* Social Sciences: `sociology`, `anthropology`, `economics`, `political-science`
* Add more subjects as needed in custom config

### 5. Content Type (custom)
* `tutorial` - Instructional content
* `reference` - Quick reference material
* `howto` - Step-by-step guides
* `api` - API documentation

## Access Control Integration

Keywords with `restrictEditing: true` integrate with the access control system:

```json
{
  "private": {
    "restrictEditing": true,
    "allowedRoles": ["admin", "author"]
  }
}
```

When a page has the `private` keyword:
* Only the original author and admins can edit
* Other users get "Access Denied" when attempting to edit
* Read access follows normal page permissions

## Schema.org Integration

User keywords can map to schema.org types for semantic web compatibility:

```json
{
  "medicine": {
    "schemaOrg": "MedicalEntity"
  },
  "tutorial": {
    "schemaOrg": "HowTo"
  },
  "api": {
    "schemaOrg": "APIReference"
  }
}
```

This enables:
* Better search engine optimization
* Semantic web integration
* Structured data markup
* Enhanced metadata for external tools

## Backend Integration

### UI Population

The backend reads `amdwiki.userKeywords` configuration to:
* Populate keyword dropdown/autocomplete in the editor
* Validate keywords during page save
* Generate keyword suggestions based on content
* Enforce access control rules

### Keyword Validation

The ValidationManager can validate user keywords:
* Check if keyword exists in configuration
* Warn about unknown keywords (but allow them)
* Enforce maximum keyword limit
* Validate access control restrictions

## Best Practices

### For Content Creators

1. **Use Descriptive Keywords**: Choose keywords that clearly describe the content
2. **Be Consistent**: Use the same keywords for similar content
3. **Don't Over-Tag**: Limit to 3-5 most relevant keywords
4. **Check Existing**: Use existing keywords before proposing new ones
5. **Combine Types**: Mix subject, status, and other categories

### For Administrators

1. **Keep It Organized**: Group keywords by category
2. **Document Purpose**: Write clear descriptions
3. **Review Regularly**: Audit keyword usage and consolidate duplicates
4. **Enable Schema.org**: Map keywords to schema.org types when possible
5. **Control Growth**: Don't let keyword list grow uncontrollably

### For Developers

1. **Use Configuration**: Always read from `amdwiki.userKeywords`
2. **Support Custom**: Allow custom config to extend defaults
3. **Validate Gracefully**: Warn but don't block unknown keywords
4. **Cache Efficiently**: Cache keyword lookups for performance
5. **Document Changes**: Update this page when adding new keywords

## Proposing New Keywords

Users with Editor role or higher can propose new keywords:

1. **Identify Need**: Determine if existing keywords don't fit
2. **Check Documentation**: Review this page for existing keywords
3. **Create Proposal**: Document the keyword purpose and usage
4. **Submit Request**: Contact admin or create a ticket
5. **Admin Reviews**: Admin adds to custom config if approved

### Proposal Template

```markdown
**Keyword**: astronomy
**Category**: subject
**Description**: Astronomy and space science content
**Reason**: We have 15 pages about astronomy with no appropriate keyword
**Schema.org Mapping**: (optional) AstronomicalObject
```

## Migration Notes

### From Old System

If migrating from a different keyword system:

1. **Map Old → New**: Create mapping of old keywords to new keywords
2. **Update Metadata**: Batch update all pages with new keywords
3. **Add Custom Config**: Add frequently-used old keywords to custom config
4. **Document Changes**: Update this page with migration details

### Keyword Consolidation

To consolidate similar keywords:

1. **Identify Duplicates**: Find similar keywords (e.g., "med" and "medicine")
2. **Choose Primary**: Select the most descriptive keyword
3. **Update Pages**: Batch replace old keywords with primary
4. **Disable Old**: Set `enabled: false` for deprecated keywords
5. **Document**: Note deprecation in keyword description
