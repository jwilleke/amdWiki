# Page Audience

The **Audience** setting controls which roles can **view** a page. It is optional — leaving all roles unchecked means global access policies apply.

## Roles

%%table-striped
|| Role || Description ||
| admin | Full system access to all features |
| user-admin | Can manage users — create, edit, deactivate, delete |
| editor | Can create, edit, delete, and rename pages |
| contributor | Can create and edit pages |
| reader | Read-only access to content |
| member | Community member — same read access as reader |
| anonymous | Public access without authentication |
/%

## How It Works

- Add one or more **role names** or individual **usernames** to restrict view access to those principals only
- Leave **empty** to apply the global access policies (the default)
- Audience entries can be **role names** (e.g. `editor`) or **usernames** (e.g. `jim`) — both are matched
- Audience does **not** grant edit access — only view access

## Per-Action Access (`access` field)

For more granular control, use the `access` frontmatter field instead of `audience`. It accepts an object keyed by action name, each with an array of principals:

```yaml
access:
  view:
    - reader
    - editor
  edit:
    - editor
```

`audience` is a shorthand for `access.view` only.

## The `private` Keyword

If a page has the `private` [User Keywords] keyword, the page is stored in a private location and the Audience field is bypassed entirely. Access is restricted to:

- Users with the `admin` role
- The page's **creator** — the user who first saved the page with the `private` keyword

> **Creator vs Author**: access uses the page's **creator** as recorded in the page index, not the `author` frontmatter field. If the `author` field differs from the actual creator, the `author` field is ignored for access control purposes.

### Applying the `private` Keyword

Add `private` to the User Keywords field in the editor, or in frontmatter:

```yaml
user-keywords:
  - private
```

### What `private` Does Not Do

- Does **not** hide the page from admins — admins always retain access
- Does **not** encrypt the page content — it is access-controlled, not encrypted
- Does **not** apply separately to attachments — attachment access follows the page
- Removing the `private` keyword restores normal access policy behaviour

### Private vs Audience

%%table-striped
|| Feature || Audience field || `private` keyword ||
| Where configured | Page metadata (Audience) | `user-keywords` frontmatter / editor |
| Who can view | Any roles or usernames listed | Admin + creator only |
| Storage location | Normal page store | Private location |
| Bypasses Audience | — | Yes — Audience setting is ignored |
/%

The `private` keyword takes precedence. If a page has both an Audience setting and the `private` keyword, the Audience setting is ignored.
